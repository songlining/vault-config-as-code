Vault Identity System

entity · entity alias · identity group · group alias

Vault's identity system unifies authentication across multiple backends into a single identity. A person who logs in via GitHub and LDAP is still one Entity with two Entity Aliases. Groups aggregate entities and carry policies — giving you one place to manage access regardless of how someone authenticates.

Entity
Canonical Identity
Represents a single person or machine. One entity per real-world identity. Carries metadata and directly-assigned policies via identity/entity.
Entity Alias
Auth Method Link
Maps a login credential to its entity. Each alias is scoped to one auth mount — e.g., github/alice or ldap/alice. One entity can have many aliases.
Identity Group
Collection of Entities
Internal groups have explicit members. External groups get members from an IdP via a Group Alias. Groups carry policies inherited by all members.
Policy
Access Rules
ACL or Sentinel policies attached to entities or groups. The effective token policy set is the union of entity policies + all group policies.
Auth Method
Login Backend
GitHub, LDAP, AWS IAM, Userpass, PKI, etc. Each mounted auth method creates aliases in its own namespace — auth/github/, auth/ldap/.

Login Resolution Flow
1
Authenticate
User logs in via an auth method — e.g., vault login -method=github. The auth method validates the credentials.
2
Resolve Alias
Vault looks up the Entity Alias for this auth mount + login name. If no alias exists, Vault auto-creates a new Entity and Alias.
3
Load Entity
The alias points to exactly one Entity. Vault loads the entity's metadata, directly-assigned policies, and group memberships.
4
Inherit Group Policies
Vault walks the entity's group memberships — including nested subgroups — and unions all group policies together.
5
Issue Token
The resulting token carries: auth method policies + entity direct policies + all inherited group policies. This is the effective policy set.
Internal Group
Members are explicitly assigned — you add entities or child groups directly in Vault. Best for Vault-native access control where you manage membership manually or via Terraform.
resource "vault_identity_group" "eng" { name = "engineering" type = "internal" policies = ["kv-read"] member_entity_ids = [ vault_identity_entity.alice.id, vault_identity_entity.bob.id, ] }
External Group
Members come from an external identity provider via a Group Alias. When the IdP says "alice is in team-platform", Vault maps that to this group automatically.
resource "vault_identity_group" "gh_eng" { name = "gh-engineering" type = "external" policies = ["kv-read"] } resource "vault_identity_group_alias" "gh_eng" { name = "engineering" mount_accessor = vault_github_auth_backend.org.accessor canonical_id = vault_identity_group.gh_eng.id }
Key rules. Each Entity Alias is scoped to exactly one auth mount — you can't share an alias across methods. An Entity can have at most one alias per auth mount. Merging two entities is permanent and combines all aliases, group memberships, and policies. External group membership is read-only in Vault — it's driven entirely by the IdP.
